## Ways to Contribute

### 1. Feedback & Discussion

The simplest way to contribute is to share your perspective:

- **Report issues** with existing specifications
- **Propose new features** or standard additions
- **Ask questions** and clarify ambiguities
- **Share use cases** from your industry or domain

Open an issue on GitHub or email us at info@icoun.org.

### 2. Documentation & Specifications

Help improve the quality and clarity of our standards:

- Fix typos, grammatical errors, and unclear language
- Add examples and use cases
- Improve diagrams and visualizations
- Review specifications for accuracy and completeness

### 3. Reference Implementations & SDKs

When we release code repositories, you can contribute by:

- Submitting bug fixes
- Adding features to SDKs
- Writing tests and improving test coverage
- Porting SDKs to additional languages
- Optimizing performance

### 4. Working Groups

Deepen your involvement by joining a Working Group:

| Working Group | Focus Area |
|---|---|
| DPP-CQ Technical WG | Core specification, data models, and protocol design |
| Governance & Compliance WG | Governance framework, legal alignment, certification |
| Interoperability WG | Integration with W3C, GS1, ISO, EU DPP, UNECE |
| Use Cases & Pilots WG | Real-world implementation, case studies, sectoral adaptation |

**To join a Working Group**, email info@icoun.org with:
- Your name and affiliation (if any)
- Area of interest and relevant background
- Expected level of participation

## Issue Guidelines

### When to Open an Issue

- Bug reports or errors in specifications
- Feature requests or standard additions
- Questions about the standard or implementation
- Discussion of design decisions
- Proposed changes or improvements

### Issue Best Practices

1. **Search first** — check existing issues before opening a new one
2. **Be specific** — provide context, examples, and relevant sections
3. **Use clear titles** — summarize the issue in one line
4. **Label appropriately** — use labels to categorize (bug, enhancement, question, etc.)

## Pull Request Process

We accept contributions via Pull Requests (PRs).

### For Documentation/Spec Changes

1. Fork the repository
2. Create a branch with a descriptive name (e.g., `fix/dpp-cq-field-typo`, `feature/add-quality-tier-definition`)
3. Make your changes
4. Ensure the document renders correctly and links work
5. Submit a PR with a clear description of:
   - What changed
   - Why the change is needed
   - Any implications for other sections

### For Code Changes (when applicable)

1. Follow the project's coding style
2. Add or update tests as needed
3. Ensure all existing tests pass
4. Update documentation for new features
5. Submit a PR with a clear description

### PR Review Process

1. All PRs require review from at least one maintainer
2. Reviewers may request changes or clarification
3. Once approved, a maintainer will merge the PR
4. Major changes to standards require Working Group review

> **Note**: For significant changes to core standards, please open an issue first to discuss the proposal before investing time in a PR.

## Coding Standards

When contributing code (reference implementations, SDKs, tools):

- Follow the language's established conventions (PEP 8 for Python, ESLint for JavaScript, etc.)
- Write clear, maintainable code with meaningful names
- Include comments for non-obvious logic
- Add tests for new functionality
- Keep PRs focused — one feature or fix per PR

## Documentation Standards

For specification and documentation contributions:

- Use clear, precise, and unambiguous language
- Define technical terms on first use
- Include examples for complex concepts
- Maintain consistent terminology across documents
- Follow the structure and formatting conventions of existing documents
- Cite relevant external standards and references

## Working Group Process

Working Groups follow the ICO standards development lifecycle:

1. **Working Draft (WD)** — Initial draft developed by the Working Group
2. **Committee Draft (CD)** — Draft approved by the Technical Committee
3. **Public Review (PR)** — Draft published for public comment
4. **Final Standard (IS)** — Final version approved by the Standards Council

Working Groups typically meet bi-weekly. Meeting minutes are published in the repository.

## Code of Conduct

We are committed to providing a welcoming, inclusive, and respectful community. All participants are expected to:

- **Be respectful** of differing viewpoints and experiences
- **Be collaborative** and work toward consensus
- **Be constructive** in criticism and feedback
- **Take responsibility** for your words and actions
- **Focus on what is best** for the community and the standards

Instances of unacceptable behavior may be reported to info@icoun.org. The Standards Council retains the right to take appropriate action, including removal from the community.

- **Documentation and specifications**: [CC BY 4.0](https://creativecommons.org/licenses/by/4.0/)
- **Software and code**: [Apache License, Version 2.0](LICENSE)

### Contributor License Agreement (CLA)

For significant code contributions (above a threshold determined by the Technical Committee), we may require a Contributor License Agreement (CLA). This ensures the project has the necessary rights to distribute your contribution. We will contact you directly if a CLA is required for your PR.
